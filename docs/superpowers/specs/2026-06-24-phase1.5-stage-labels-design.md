# Phase 1.5 Data Task — Historical group/knockout stage labels

_Design spec. Decision delegated to the implementer, grounded in Luo et al. 2025 + the project's zero-hallucination / reproducibility ethos. Produces the data that unblocks the stage-aware regime modeling recorded as Phase 1.5 in the model-quality spec (2026-06-23). Scope here = the **data task** (labels + derivation + validation) only; wiring stage into the harness/model is a separate follow-up._

## Context

Phase 1 (PR #35) ended in a HOLD and surfaced the real next lever: the documented draw-blindness is a **conditional** (high-favorite knockout) failure, not an aggregate one. Luo et al. (2025) confirm this independently — adding a binary **stage** identifier (`group=1`, `knockout=2`) was the single largest predictive gain in their FIFA World Cup model, working by reallocating probability away from draws in knockout play. To test a stage-aware regime in our walk-forward harness we need a **group/knockout label per historical match**, which `data/raw/results.csv` does not carry (columns: `date, home_team, away_team, home_score, away_score, tournament, city, country, neutral`).

## Problem statement

Produce a reproducible, zero-hallucination **binary group/knockout label** for every 1990+ finals-tournament match (FIFA World Cup, UEFA Euro, Copa América, African Cup of Nations, AFC Asian Cup), derived **structurally from `results.csv` itself** — no hand-encoded edition facts (hallucination risk) and no external fetch dependency (fragility) — validated by bracket invariants, and emitted as a join-ready artifact.

## Goals

1. **Binary label** (`group` | `knockout`) per match — the granularity Luo et al. validated as the predictive lever. No finer rounds.
2. **Structural derivation from `results.csv` only** — deterministic, reproducible, auditable; asserts no edition-specific facts from memory or the web.
3. **Validated** by bracket invariants (below); editions that do not cleanly resolve are **flagged, not fabricated**.
4. **Join-ready artifact** `data/stage-labels.json` keyed by match identity, consumable by the trainer/harness in a follow-up.
5. **Scope 1990+** (the harness `EVAL_FROM` regime), with the derivation general enough to extend earlier later.

## Non-goals

- Wiring stage into the model/harness (stage-aware regime) — separate follow-up once labels exist.
- Finer round labels (R16/QF/SF/F) — YAGNI; binary is the validated lever.
- Pre-1990 editions (older formats: round-robin final groups in 1950, etc.) — out of scope; the algorithm may flag them as unresolved.

## Approach — backward single-elimination bracket reconstruction

Within any modern (1990+) edition, the group stage (round-robin) fully precedes the knockout stage (single elimination), and the knockout rounds **halve** in size: Final = 1 match, Semifinals = 2, Quarterfinals = 4, Round-of-16 = 8, (Round-of-32 = 16 for the 2026 48-team format). A third-place playoff, when present, sits parallel to the final (the two semifinal losers). These structural facts — not edition-specific counts — drive the derivation.

For each `(tournament, year)` edition, over its matches sorted by `(date, original-order)`:

1. **Final:** the chronologically last match of the edition is the Final (round size 1).
2. **Backward round-doubling:** the matches feeding round *r* are those whose **winner** appears as a participant in a round-*r* match and whose **loser does not reappear later in the edition** (single-elimination property). A valid feeding round must contain exactly **double** the prior round's match count (1 → 2 → 4 → 8 → …). Continue while the doubling and single-elimination properties hold.
3. **Boundary:** stop when the next candidate layer fails the doubling/elimination invariants — that is the group/knockout boundary. Everything in the reconstructed bracket is `knockout`; everything earlier in the edition is `group`.
4. **Third-place playoff:** detect a match between the two semifinal losers dated on/near the Final; label it `knockout` (it is single-elimination class) and exclude it from the doubling chain.
5. **Unresolved editions:** if the suffix does not reconstruct into a clean halving bracket (e.g., pre-1990 round-robin finals, data gaps), do **not** guess — record the edition in an `unresolved` list with the reason; its matches receive no label (and are simply absent from any stage-aware split downstream).

This is purely a function of the recorded results; re-running it on the same `results.csv` yields identical labels.

## Components & interfaces

### New: `lib/stage-derivation.ts` (pure)
- `type EditionMatch = { date: string; home: string; away: string; hs: number; as: number; idx: number }`
- `type StageLabel = "group" | "knockout"`
- `deriveEditionStages(matches: EditionMatch[]): { labels: Map<number, StageLabel>; resolved: boolean; reason?: string }` — the bracket reconstruction for one edition; `resolved:false` + `reason` when invariants fail.
- `validateBracket(knockoutMatches): { ok: boolean; reason?: string }` — checks single-final, round-doubling, single-elimination, optional 3rd-place.
- Pure, no I/O. Reuses `isFinalsTournament` from `lib/validation` for filtering at the call site.

### New: `scripts/derive-stage-labels.mts` + `npm run ml:stage-labels`
- Reads `data/raw/results.csv` (read-only), groups finals-tournament matches by `(tournament, year)` for `year >= 1990`, runs `deriveEditionStages` per edition, and writes `data/stage-labels.json`:
  ```json
  { "generatedFrom": "data/raw/results.csv", "evalFrom": "1990",
    "labels": [{ "date","home","away","tournament","stage" }, ...],
    "unresolved": [{ "tournament","year","reason" }],
    "summary": { "editions": N, "resolved": M, "group": G, "knockout": K } }
  ```
- Prints a summary and the unresolved list. Manual/offline; **not** a commit gate. Data-safe (read-only on results.csv; writes only `data/stage-labels.json`).

### Validation harness (in the script + tests)
- Per-edition invariants: exactly one Final; knockout count is `2^r - 1` (+1 if 3rd-place); every knockout loser (except 3rd-place participants) absent from later matches; all `group`-labeled matches precede all `knockout`-labeled matches by date.
- A coverage report: how many 1990+ editions resolved vs flagged.

## Testing (TDD)

- `tests/stage-derivation.test.ts` (synthetic fixtures — **no real-edition facts asserted from memory**):
  - A constructed 8-team single-elim bracket (QF→SF→F) + a round-robin group prefix → algorithm labels exactly the bracket as knockout, group prefix as group.
  - With a third-place playoff added → labeled knockout, bracket still validates.
  - A final-group-round match whose loser is eliminated (doesn't reappear) is NOT mislabeled knockout (the doubling invariant excludes it).
  - A degenerate/round-robin-final edition → `resolved:false` with a reason (not fabricated labels).
  - `validateBracket` accepts a clean 1/2/4 tree and rejects a non-doubling set.
- The full suite stays green.

## Gates before commit

`npm test` · `npm run lint` · `npm run design:inspect` · `npm run inspect:execution` · `npm run model:inspect` · `npm run build`. `ml:stage-labels` is run manually to produce the artifact (offline, not a commit gate).

## Risks & honest failure modes

- **Some editions may not resolve** (data quirks, missing matches, atypical formats). That is acceptable and explicit (`unresolved` list) — better than fabricated labels. The coverage report makes the gap visible.
- **Same-day boundary matches** could perturb ordering; the `(date, idx)` sort plus the structural (not purely chronological) reconstruction mitigates this.
- **Third-place detection** is heuristic (SF losers, near-final date); if ambiguous, the match stays `knockout` only when it satisfies the SF-loser property, else the edition is flagged rather than mislabeled.
